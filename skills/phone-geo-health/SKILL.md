# 手机GEO健康检查技能

本技能定义了手机产品在豆包 AI 中的 GEO 健康检查完整工作流，核心目标是验证AI对手机产品的认知和推荐是否与官方市场团队打造的产品定位一致。

***


## Step 1: 产品画像构建与用户确认（F1）

### 目的

获取用户产品请求，构建目标手机产品的完整画像并请用户确认，作为后续Query生成和认知一致性分析的基准。

### 执行流程

01. **识别用户请求**：用户输入如"分析下华为pocket"，自动识别手机型号，如果用户没有说明手机型号，请向用户询问；
02. **构建初始画像**：基于手机型号生成完整产品画像，可以联网搜索或自己补全
```
产品画像 — {产品名称}

▎基础信息
- 产品名称：{完整型号}
- 品牌：{品牌}
- 价格段：{如"7000-9000元高端旗舰"}
- 品类：{如"小折叠屏手机"}

▎市场定位
- 定位描述：{官方定位}
- 目标人群：{人群描述}
- 核心场景：{主要使用场景}

▎核心卖点（2-4个）
1. {卖点1}：{官方描述}
2. {卖点2}：{官方描述}
3. {卖点3}：{官方描述}

▎竞品定位
1. {竞品1}：{竞争关系}
2. {竞品2}：{竞争关系}

▎风险信号
1. {潜在弱点1}
2. {潜在弱点2}
```

03. **呈现给用户确认**：展示画像并询问是否确认或修改
```
以上是为您构建的「{产品名称}」产品画像，请您确认：

✅ **信息准确，继续检查** → 直接回复"确认"
📝 **需要修改** → 告诉我哪些信息需要调整
```

04. **等待用户确认**：用户确认后再进入Step 2决策阶段框架



## Step 2: Query生成与决策阶段框架（F2）

### 目的

基于产品画像，生成覆盖消费者决策全链路的测试Query，验证AI对产品的认知是否与市场定位一致。

### 消费者决策三阶段框架

阶段1：认知/推荐，用户处于需求发现阶段，寻求初步推荐。

| 细分场景 | 典型Query示例 | 验证目标 |
|---------|-------------|---------|
| 泛推荐 | "帮我推荐一款手机" | 产品在无明确条件下的自然曝光 |
| 价位推荐 | "7000元左右手机推荐" | 价格定位认知是否准确 |
| 人群推荐 | "女生用什么手机好" | 目标人群匹配度 |
| 场景推荐 | "拍照好的手机推荐" | 核心卖点关联 |

阶段2：对比，用户进入产品比较阶段，评估不同选项。

| 细分场景 | 典型Query示例 | 验证目标 |
|---------|-------------|---------|
| 产品对比 | "华为Pocket 2和三星Z Flip6哪个好" | 直接竞争中的胜负率、对比维度、优劣势 |

阶段3：决策，用户接近购买决策，关注风险和价值。

| 细分场景 | 典型Query示例 | 验证目标 |
|---------|-------------|---------|
| 值不值得买 | "华为Pocket 2值得买吗" | 购买建议倾向 |
| 风险疑虑 | "华为Pocket 2发热严重吗" | 风险点认知 |
| 使用体验 | "华为Pocket 2使用体验怎么样" | 综合评价 |

### Query生成
以细分场景作为Query类型（7种），生成query
- **默认 Query 数量**：10个
- **阶段覆盖要求**：3个决策阶段至少各有2个Query

#### Query类型与对应阶段

| Query类型（7种） | 对应阶段 | 说明 | 示例 |
|-----------|---------|------|------|
| 泛推荐型 | 认知/推荐 | 无条件泛推荐 | "帮我推荐一款手机" |
| 价位推荐型 | 认知/推荐 | 按价格区间推荐 | "7000-9000元手机推荐" |
| 人群推荐型 | 认知/推荐 | 按人群推荐 | "女生用什么手机好" |
| 场景推荐型 | 认知/推荐 | 按使用场景推荐 | "拍照好的小折叠推荐" |
| 产品对比型 | 对比 | 直接产品对比（与核心竞品） | "Pocket 2和Z Flip6哪个好" |
| 购买决策型 | 决策 | 值不值得买 | "Pocket 2值得买吗" |
| 风险疑虑型 | 决策 | 产品风险咨询 | "Pocket 2续航怎么样" |

### Query分配策略

| 决策阶段 | Query数 | Query类型 |
|---------|--------|-----------|
| 认知/推荐 | 5 | 泛推荐+价位+人群+场景×2 |
| 对比 | 2 | 产品对比×2（与不同竞品） |
| 决策 | 3 | 购买决策+2个风险疑虑 |

### 内部输出格式

> 以下为内部数据结构，用于后续步骤，不直接展示给用户

```json
[
  {
    "query": "7000-9000元手机推荐",
    "type": "价位推荐型",
    "stage": "认知/推荐",
    "dimension": "预算",
    "expected_position": "高端旗舰"
  }
]
```

### 生成规则

1. **自然语言**：真实用户会问的自然语言，不是关键词堆砌
2. **按阶段命名限制**：
   - 认知/推荐阶段：❌ 禁止提及本产品名称，测试AI主动推荐能力
   - 对比阶段：✅ 必须包含目标产品与竞品的对比
   - 决策阶段：✅ 可提及本产品名称
3. **匹配产品画像**：Query应覆盖产品的核心卖点和价格定位
4. **覆盖决策阶段**：确保3个阶段都有测试
5. **竞品选择**：对比型Query中，选择产品画像中定义的竞品进行对比

## Step 3: 豆包问答执行（F3）

### 目的

使用doubao-chat向豆包发送测试Query，收集回答数据。

### 调用方式

**方式一：命令行直接传入（适合少量Query）**
```bash
python skills/doubao-chat/scripts/chat.py "Q1" "Q2" "Q3" --standalone --timeout=180
```

**方式二：从文件读取（推荐，适合批量Query）**
```bash
# 1. 将Query写入文件（UTF-8编码）
echo "Q1" > queries.txt
echo "Q2" >> queries.txt

# 2. 执行（Windows需处理编码问题）
# Windows PowerShell:
$env:PYTHONIOENCODING="utf-8"; python skills/doubao-chat/scripts/chat.py --questions-file queries.txt "placeholder" --standalone --timeout=180

# Windows CMD:
chcp 65001
set PYTHONIOENCODING=utf-8
python skills/doubao-chat/scripts/chat.py --questions-file queries.txt "placeholder" --standalone --timeout=180

# Linux/Mac:
PYTHONIOENCODING=utf-8 python skills/doubao-chat/scripts/chat.py --questions-file queries.txt "placeholder" --standalone --timeout=180
```

**参数说明：**
| 参数 | 必填 | 说明 |
|------|------|------|
| `--questions-file` | 否 | 从文件读取Query列表，每行一个 |
| `questions` | 是 | 命令行传入的Query（至少一个，可为占位符） |
| `--standalone` | 否 | 使用Playwright内置Chromium（推荐） |
| `--timeout` | 否 | 每个Query等待回答的最大时间（秒），默认60 |
| `--new-thread` | 否 | 每个Query新开聊天窗口 |

### 执行前检查

1. 确认Cookie有效
2. 告知用户预估耗时（10个Query约10-15分钟）

### 常见问题与解决

| 问题 | 现象 | 原因 | 解决 |
|------|------|------|------|
| **编码乱码** | 回答内容显示为乱码（如"һ�������ᱡ�콢"） | Windows终端默认GBK编码，与UTF-8输出冲突 | 设置环境变量 `PYTHONIOENCODING=utf-8`，并确保终端为UTF-8编码（`chcp 65001`） |
| **参数解析失败** | 报错 `error: the following arguments are required: questions` | `--questions-file`需要配合positional参数 | 添加占位符参数，如 `"placeholder"` |
| **执行超时** | Query发送后长时间无响应 | 豆包回答较慢，默认60秒不足 | 增加 `--timeout=180`（3分钟） |
| **UnicodeEncodeError** | 报错 `'gbk' codec can't encode character` | Python在Windows输出UTF-8内容时编码失败 | 设置 `PYTHONIOENCODING=utf-8` |

> 以下为内部数据结构，用于后续步骤，不直接展示给用户

参考原格式，包含question、answer、references等字段。

## Step 5: 报告生成（F5）

### 目的

生成完整的健康检查报告文件，给用户。

### 报告结构

1. **检查摘要**：认知一致性得分、核心发现
2. **产品画像**：输入的产品定位信息
3. **认知/推荐阶段分析**：泛推荐、价位、人群、场景
4. **对比阶段分析**：胜负率、对比维度、优劣势
5. **决策阶段分析**：购买决策、风险疑虑
6. **引用来源分析**：豆包引用情况、优化方向
7. **认知一致性评估**：各维度匹配度评分
8. **优化建议**：针对认知偏差的改进方向

### 输出方式

Markdown格式，文件名：`{产品名称}_GEO健康检查报告_{日期}.md`

## 完整工作流

```
用户输入产品请求（如"分析下华为pocket"）
      │
      ▼
Step 1: 识别手机型号 + 构建产品画像
      │
      ▼
Step 1.5: **用户确认** → 展示画像，等待用户确认或修改
      │
      ▼（用户确认后）
Step 2: Query生成（覆盖3阶段，验证认知一致性）
      │
      ▼
Step 3: 豆包问答执行
      │
      ▼
Step 4: 认知与表现分析（按决策阶段评估+引用分析）
      │
      ▼
Step 5: 生成健康检查报告
```
